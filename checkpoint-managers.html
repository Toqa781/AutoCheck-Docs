Checkpoint Managers

Framework-specific implementations that handle atomic file I/O for saving and loading complete training state.

Relevant source files mlTrainingModel.py

A common interface, framework-specific bodies

Every CheckpointManager implementation exposes the same two operations, save() and load(), so the controller never needs to know which framework it's supervising. The differences live entirely inside each implementation.

ImplementationFrameworkCaptures
PyTorchCheckpointManagerPyTorchModel state dict, optimizer state dict, scheduler state, custom metadata.
KerasCheckpointManagerTensorFlow / KerasModel weights, optimizer config, training metadata.
SKLearnCheckpointManagerScikit-LearnFitted estimator state via pickling, plus run metadata.
GenericCheckpointManagerFramework-agnosticAny registered user-defined objects that don't fit a specific framework manager.

Structured storage layout

All managers write into the same hierarchical directory structure, keeping checkpoints from different stages and steps cleanly separated and easy to locate for resume.

text
checkpoints/
└── stage/
    └── epoch-batch/
        ├── model_state.pt
        ├── optimizer_state.pt
        └── metadata.json
Atomic writes

Each manager writes to a temporary path and renames into place once complete, so a crash mid-write never leaves a corrupted checkpoint that resume could mistakenly load.
