FAQ

Quick answers to the questions that come up most when getting started with AutoCheck.

General

Do I need to modify my code to use AutoCheck?

No. AutoCheck is zero-code-modification — you point it at your unmodified Python script, Java program, or compiled C/C++ binary, and it observes and checkpoints it from the outside. You don't import anything, call a save function, or add any AutoCheck-specific lines to your program.

Which languages does AutoCheck support?

Python, Java, C, and C++. See Language Support for what each one requires and the worked examples for a complete config per language.

What's the difference between ML mode and HPC mode?

execution_mode: ml is for PyTorch or TensorFlow training jobs — AutoCheck automatically discovers your model, optimizer, and training progress. execution_mode: hpc is for everything else (Java, C, C++, or general long-running simulations) — you tell AutoCheck which variables to track via hpc.tracked_states. See Which config do I need? for the full decision table.

Configuration

Where do I put config.yaml?

In your project's working directory, then run AutoCheck from that same location — or pass an explicit path with --config /path/to/my_config.yaml. See Where to place the file.

Can I check my config without actually starting the job?

Yes — use autocheck run --config config.yaml --validate-only. This checks your YAML for syntax and logical errors and exits without launching your program.

Which config keys are actually required?

system.execution_mode, system.language, checkpoint.save_dir, and checkpoint.program_path are required for every job. hpc.tracked_states is required for HPC jobs specifically. Everything else has a sensible default. See Configuration Schema Reference for the full list.

How do I change how often checkpoints are saved?

Set checkpoint.interval in seconds — for example, interval: 300 checkpoints every 5 minutes. Pair it with checkpoint.max_session_time to also cap the total run length, useful for matching an HPC scheduler's wall-clock limit.

Running jobs

How do I stop a running job?

autocheck stop --config config.yaml. This sends a graceful stop request — AutoCheck saves one final checkpoint before exiting, so you don't lose progress made since the last automatic save.

How do I resume a job after it stops?

autocheck resume --config config.yaml. AutoCheck automatically finds the most recent valid checkpoint in save_dir and continues from there. See Resume Jobs for resuming from a specific checkpoint instead of the latest one.

Why does resume fail with an error about a running job?

AutoCheck only allows one job per save_dir at a time, to prevent two processes from writing checkpoints to the same directory. If resume refuses to start, make sure the original job has actually stopped — check for a stale .autocheck.pid file in your save_dir if you're sure nothing is running.

Can I start a fresh run without overwriting my existing checkpoints?

Yes — use --force-fresh with run. AutoCheck creates an isolated subdirectory instead of writing into the same save_dir, so previous checkpoints stay untouched.

Checkpoints and storage

What exactly gets saved in a checkpoint?

For ML jobs: model weights, optimizer state, and training progress. For HPC jobs: whatever variables you listed under hpc.tracked_states. See Snapshots for the full breakdown by framework.

How many checkpoints does AutoCheck keep?

The number set by checkpoint.keep_last — the default is 3. Older checkpoints beyond that are deleted automatically as new ones are saved, to keep disk usage under control on long runs.

Can I get notified when a checkpoint is saved or a job fails?

Yes — add a notify section to your config with your email and on_checkpoint / on_failure set to true. See the notify section of the Configuration Schema Reference.
