Checkpoint Loop and Trigger Logic

The exact conditions that cause AutoCheck to save: time-based intervals, session limits, and step-based boundaries.

Relevant source files main.py config.yaml

Trigger types

Two independent policies can fire a checkpoint. Either is sufficient on its own, and both can be active simultaneously.

TriggerConfig keyFires when
Interval triggerinterval_minutesElapsed wall-clock time since the last checkpoint reaches the configured value.
Session limit triggersession_time_limitTotal elapsed session time reaches the configured value — the job stops and saves a final checkpoint.
Signal triggern/aAn intercepted SIGTERM or SIGUSR1 requests an immediate final save.

Example timeline

With interval_minutes: 30 and session_time_limit: 120, a single session produces a predictable save cadence:

text
t=00:30  →  checkpoint saved (interval trigger)
t=01:00  →  checkpoint saved (interval trigger)
t=01:30  →  checkpoint saved (interval trigger)
t=02:00  →  session_time_limit reached, final checkpoint saved, job stops
Set session_time_limit below your scheduler's limit

Leave enough margin for the final checkpoint write itself to complete before the HPC scheduler forcibly kills the process.
