Checkpoint Loop and Trigger Logic

The exact conditions that cause AutoCheck to save: time-based intervals, session limits, and step-based boundaries.

Trigger types

Two independent policies can fire a checkpoint. Either is sufficient on its own, and both can be active simultaneously.

TriggerConfig keyFires when
Interval triggerinterval_minutesElapsed wall-clock time since the last checkpoint reaches the configured value.
Session limit triggersession_time_limitTotal elapsed session time reaches the configured value — the job stops and saves a final checkpoint.
Signal triggern/aAn intercepted SIGTERM or SIGUSR1 requests an immediate final save.

Example timeline

With interval_minutes: 30 and session_time_limit: 120, a single session produces a predictable save cadence:

text
t=00:30  →  checkpoint saved (interval trigger)
t=01:00  →  checkpoint saved (interval trigger)
t=01:30  →  checkpoint saved (interval trigger)
t=02:00  →  session_time_limit reached, final checkpoint saved, job stops
Set session_time_limit below your scheduler's limit

Leave enough margin for the final checkpoint write itself to complete before the HPC scheduler forcibly kills the process.

Checkpoint Managers

Framework-and-Language specific implementations that handle atomic file I/O for saving and loading complete training state.

A common interface, framework-specific bodies

Every CheckpointManager implementation exposes the same two operations, save() and load(), so the controller never needs to know which framework it's supervising. The differences live entirely inside each implementation.

ImplementationFrameworkCaptures
PyTorchCheckpointManagerPyTorchModel state dict, optimizer state dict, scheduler state, custom metadata.
KerasCheckpointManagerTensorFlow / KerasModel weights, optimizer config, training metadata.
SKLearnCheckpointManagerScikit-LearnFitted estimator state via pickling, plus run metadata.
GenericCheckpointManagerFramework-agnosticAny registered user-defined objects that don't fit a specific framework manager.

Structured storage layout

All managers write into the same hierarchical directory structure, keeping checkpoints from different stages and steps cleanly separated and easy to locate for resume.

text
checkpoints/
└── stage/
    └── epoch-batch/
        ├── model_state.pt
        ├── optimizer_state.pt
        └── metadata.json
Atomic writes

Each manager writes to a temporary path and renames into place once complete, so a crash mid-write never leaves a corrupted checkpoint that resume could mistakenly load.
