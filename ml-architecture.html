ML Training Architecture

How AutoCheck wraps an unmodified ML training script, intercepts its variables in real time, and persists model state across PyTorch, Keras, and Scikit-Learn without a single import in user code.

Relevant source files stateTracker/MLStateTracker.py checkpointManager/PyTorchCheckpointManager.py checkpointManager/KerasCheckpointManager.py layers/TraceLayer.py Testing_phase/mlTrainingModel.py Testing_phase/mlTrainingKerasModel.py

Overview

AutoCheck operates as a sidecar to the user's training script. The user never modifies their training code — they simply invoke AutoCheck via the CLI and point it at their script. From that moment, AutoCheck manages the entire checkpoint lifecycle: it detects the training framework in use, instruments the running process via sys.settrace, and periodically serializes the complete training state to disk.

This page covers the ML-specific half of that architecture. For the HPC variant (signal-based, iteration-counter-driven) see HPC Architecture.

Zero-code-modification guarantee

Every example on this page — PyTorch, Keras, and Scikit-Learn — was run against scripts that contain no AutoCheck imports and no explicit save calls. State is captured entirely through the TraceLayer inspection mechanism.

Component map

Five components work in concert for every ML checkpointing run. The table below names each one, maps it to its code entity, and summarises its single responsibility.

RoleCode entityResponsibility
The Brain AutonomousController Decides when to save, load, or stop based on epoch/step triggers defined in config.
The Bridge RunnerScript Entry point. Bootstraps the user's script in a daemon thread and starts the controller loop.
The Monitor MLStateTracker Holds the live ground-truth of the training run: epoch, global_step, model weights, optimizer state.
The Inspector TraceLayer Uses sys.settrace to read local variables out of the user script without any code changes.
The Serialiser PyTorchCheckpointManager / KerasCheckpointManager Handles atomic writes and checkpoint rotation for the specific ML framework in use.

Execution flow

MLStateTracker

MLStateTracker is the source of truth for a running ML job. It inherits from BaseTracker and splits the variables it manages into two lists with very different access costs.

Poll list vs. target list

The controller needs to check whether a checkpoint trigger has fired on every loop iteration — potentially thousands of times per minute. Deserialising a full model weights tensor on every check would be prohibitively expensive. The split below solves that:

ListFieldsAccess costUsed for
Poll list epoch, global_step, batch_idx Cheap — scalar integers Evaluating trigger conditions (epoch % interval == 0)
Target list model, optimizer, scheduler, rng_state, amp Expensive — full tensor dicts Actual checkpoint payload written to disk

Key methods

update_ckpt_method()

Calls provider.fetch_ckpt() to retrieve only the poll list. The AutonomousController calls this on every loop tick. It validates that the returned dictionary contains all required integer fields and raises TypeError immediately if any are missing — so misconfigured scripts fail fast rather than silently skipping checkpoints.

update_all_from_prov()

Calls provider.fetch_all(), which triggers the TraceLayer to capture state_dict() from the model and optimizer objects inside the running user script. This is only called immediately before writing a checkpoint, keeping the expensive path off the hot loop.

snapshot()

Calls update_all_from_prov() then packages the training state into a single dictionary that includes metadata alongside the weights:

python 
# Fields included in every ML snapshot dict
{
    "run_id":             str,    # unique ID for this run
    "method":            "ml",
    "start_time":        float,  # Unix timestamp of job start
    "epoch":             int,
    "global_step":       int,
    "batch_idx":         int,
    "model_state_dict":  dict,  # result of model.state_dict()
    "optimizer_state":   dict,  # result of optimizer.state_dict()
    "scheduler_state":   dict,  # optional, None if not present
    "rng_state":         bytes, # torch.get_rng_state()
    "amp":               dict,  # AMP scaler state, None if not used
}

set_all_from_ckpt()

Restores state in two steps: first it calls provider.restore(state) to inject values back into the running script's memory space (so the training loop resumes at the right epoch without any code from the user), then it updates the tracker's own attributes to match.

Validation at construction time

The validate() call in the constructor checks that the environment exposes the required variables. If your training script doesn't define an epoch loop variable, the tracker raises a KeyError before the first checkpoint attempt — not silently later.

Framework support

AutoCheck ships three CheckpointManager implementations for ML workloads. The correct one is selected automatically based on the objects detected in the user script's scope.

PyTorch — PyTorchCheckpointManager

The PyTorch manager uses torch.save / torch.load and is the most fully featured of the three. It is designed around three guarantees: atomic writes (so a crash during save never corrupts the previous checkpoint), device-agnostic loading (a GPU checkpoint can be loaded on CPU and vice versa), and automatic checkpoint rotation so the storage directory doesn't grow unboundedly.

File naming convention

Checkpoints are named checkpoint_00000500.pt (8-digit zero-padded global step). The manager scans the directory with a compiled regular expression and sorts by the extracted integer — no reliance on filesystem modification times.

python 
# Pattern used to discover checkpoint files
_CKPT_RE = re.compile(r"^checkpoint_(\d+)\.pt$")

# Atomic save: write to temp file, then os.replace()
def _atomic_save(self, payload, path):
    tmp = path + ".tmp"
    torch.save(payload, tmp)
    os.replace(tmp, path)   # atomic on POSIX, near-atomic on Windows

Two interaction interfaces

The manager exposes two distinct API surfaces. Internal code (the controller) always uses the abstract contract interface; the high-level helpers exist for manual or test use.

InterfaceMethodsArgument typesUsed by
High-level helper save(), load() Explicit nn.Module, optim.Optimizer Manual use, tests, RunnerScript

Keras / TensorFlow — KerasCheckpointManager

The Keras manager handles tf.train.Checkpoint objects transparently. It is specifically designed for the more demanding manual gradient-tape loop pattern — unlike model.fit(), a gradient-tape loop requires capturing optimizer state and global_step mid-loop, which the standard Keras callbacks cannot reach.

Resume logic: when restoring, the manager provides batch_idx back to the user script's local scope, and the script's native skip logic (if batch_idx < resume_from: continue) picks up exactly where training left off. No custom callback or dataset re-winding is required.

Why manual gradient-tape?

model.fit() hides the epoch/batch loop inside Keras internals, making the loop variables invisible to TraceLayer. AutoCheck therefore requires — and strongly encourages — the explicit training loop pattern for Keras jobs where mid-epoch resumption is needed.

Scikit-Learn — SKLearnCheckpointManager

Scikit-Learn models are not iterative in the same sense as neural networks, so the manager serialises the entire model object via joblib. The expected checkpoint dict structure is:

python 
# SKLearnCheckpointManager expected payload
{
    "model":   model,    # fitted sklearn estimator or Pipeline
    "scaler":  scaler,   # optional preprocessor (StandardScaler, etc.)
    "X_train": X_train,  # training data split, for later eval
    "X_test":  X_test,
    "y_train": y_train,
    "y_test":  y_test,
}

TraceLayer — variable capture

The TraceLayer is what makes zero-code-modification possible. Python's sys.settrace hook fires on every function call, line execution, and return inside the traced thread. AutoCheck installs a trace function when the user's script thread starts and uses it to maintain a live snapshot of the script's local variable namespace.

When the controller calls provider.fetch_ckpt(), the trace layer reads the scalar integers (epoch, global_step, batch_idx) directly from the most recently captured locals. When provider.fetch_all() is called, it additionally calls .state_dict() on any discovered nn.Module and Optimizer objects — without the user script doing anything.

Performance note

sys.settrace adds a small overhead to every executed line in the traced thread. In practice this is negligible compared to GPU computation, but for CPU-bound tight loops the overhead may be measurable. The stress-test script (ml_user.py, 1 000 epochs × 10 000 iterations) showed no significant throughput regression in testing.

Testing examples

The Testing_phase/ directory ships four scripts that cover the full range of ML patterns AutoCheck supports:

PyTorch MNIST (mlTrainingModel.py)

A real CNN (two conv layers + two fully-connected layers) trained on the MNIST digit dataset via torchvision. This is the canonical integration test: it exercises model, optimizer, and scheduler checkpointing together with a DataLoader pipeline.

python 
# Variables that AutoCheck captures from this script — user writes nothing special
model     = CNN().to(device)          # nn.Module
optimizer = optim.Adam(model.parameters())
scheduler = lr_scheduler.StepLR(optimizer, step_size=5)
epoch     = 0                           # poll list
global_step = 0                        # poll list
batch_idx = 0                          # poll list

Keras MNIST (mlTrainingKerasModel.py)

A Sequential model with Conv2D and MaxPooling2D layers, trained via a manual tf.GradientTape loop. Demonstrates mid-epoch resumption: after restore, the script skips batches with if batch_idx < resume_from: continue.

Scikit-Learn Iris (model_sklearn.py)

A StandardScaler + RandomForestClassifier pipeline on the Iris dataset. Shows the underlying dictionary structure the SKLearnCheckpointManager expects and how joblib serialises the entire fitted estimator.

Minimal stress-test (ml_user.py)

A stripped-down nn.Linear(10, 2) model running 1 000 epochs × 10 000 iterations. Used to verify that TraceLayer can intercept inputs, outputs, and loss in a tight numerical loop without degrading throughput.

Running an ML job

The commands below assume AutoCheck is installed and a config.yaml exists in the working directory. See Configuration for the full schema.

bash 
# Start a new training run
autocheck run --script train.py --mode ml --config config.yaml

# Resume from the latest checkpoint
autocheck resume --script train.py --mode ml --config config.yaml

# Stop a running job gracefully (flushes a final checkpoint first)
autocheck stop --pid <PID>

Checkpoints land in the directory specified by storage_path in your config, under ml/checkpoint_XXXXXXXX.pt. The manager keeps the max_to_keep most recent files and prunes older ones automatically.
