Logging & Observability

A single structured log stream tells the complete story of every run: when checkpoints fired, why, and whether they succeeded. This page explains how to read it and what each entry means.

Logging infrastructure

A shared, structured logger is used consistently across AutonomousController, SignalLayer, JavaLayer, and every CheckpointManager implementation. All output goes to stdout by default and can be redirected to a file or piped into any standard log aggregator.

Log format

Each line follows the pattern:

text
[YYYY-MM-DD HH:MM:SS] LEVEL  message — key=value key=value …

Log levels

LevelMeaningExamples
INFONormal operating events.Checkpoint saved, job resumed, session started.
WARNUnexpected but recoverable condition.Orphaned JVM port detected and cleared.
ERRORAn operation failed. The job may stop.Checkpoint write failed, port still occupied after cleanup.

Errors & Exceptions

Two small but load-bearing modules underpin the rest of the system.

Exception types in error.py

ExceptionRaised whenWhat to do
ConfigErrorA required key is missing, a value is the wrong type, or a Java port is out of range or conflicting.Check config.yaml against the Schema Reference.
CheckpointWriteErrorA checkpoint could not be written to storage_path — disk full, permissions issue, or I/O error.Verify the path is writable and has sufficient space.
CheckpointLoadErrorA resume was attempted but the checkpoint directory is missing, corrupted, or incompatible with the current config.Confirm the checkpoint path exists and the config matches the original run.
PortErrorA Java JDWP or control port could not be cleared after the automatic cleanup attempt.Manually kill the process on the conflicting port, or remap to a different port in config.yaml.
