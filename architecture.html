Supervision chain

AutoCheck's defining trait is that supervision happens from outside the training process's normal control flow. The entry point starts an AutonomousController on the main thread; that controller then launches the user's actual training script on a separate daemon thread. From that point on, two things run concurrently: your code, executing exactly as written, and the controller, watching a shared StateTracker for signals about when to act.

Supervision chain from process entry to low-level state capture and signal interception.

ML mode vs. HPC mode

AutoCheck supports two job shapes that share the same controller but diverge at the state-tracking layer.

HPC time limits are a feature, not an edge case

The SignalLayer exists specifically because HPC schedulers send SIGTERM shortly before killing a job for exceeding its allocation. AutoCheck treats that signal as a final-checkpoint trigger rather than a crash.

Threading model in practice

Because the controller and the user script run on different threads, AutoCheck never needs cooperative yielding from your training code. The controller polls StateTracker on its own cadence, decides independently whether a save is due, and pauses the daemon thread only for the duration of the actual checkpoint write.
