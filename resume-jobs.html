Resume Jobs

If a job stops — whether you stopped it, the HPC scheduler killed it, or it crashed — AutoCheck can pick up exactly where it left off using the last saved checkpoint.

Resuming a job

Run the resume subcommand with the same config file you used for the original run:

bash
autocheck resume --config config.yaml

AutoCheck automatically finds the most recent valid checkpoint in checkpoint.save_dir and restores from it — model weights, optimizer state, and training progress for ML jobs, or your tracked variables for HPC jobs. Your program then continues running from that point as if it had never stopped.

Resuming from a specific checkpoint

By default, resume picks the latest checkpoint. If you want to roll back to an earlier one instead, pass its path explicitly:

bash
autocheck resume --config config.yaml --checkpoint checkpoints/DeepNN/checkpoint_000500.pt

This is useful if a more recent checkpoint turns out to be corrupted, or if you deliberately want to re-run training from an earlier point — for example, to try a different code change from the same starting state.

Overriding config values at resume time

The same global flags available on run also work on resume, so you can adjust a setting without editing your YAML file:

FlagShorthandWhat it overrides
--config-cPath to the YAML config file. Required.
--mode—system.execution_mode (ml or hpc) for this run only.
--save-dir-scheckpoint.save_dir for this run only.
--checkpoint—Restores from this checkpoint instead of the latest one found in save_dir.

Things to know before you resume

One job at a time per save_dir

If a job is still running against the same save_dir — detected via its PID file — resume will refuse to start. This prevents two processes from writing checkpoints to the same directory at once. Make sure the original job has actually stopped before resuming.

SituationWhat happens
No checkpoints exist yet in save_dirresume fails — there is nothing to restore from. Use run to start fresh.
A job is already running against the same save_dirresume fails immediately to avoid two processes writing to the same state.
The checkpoint passed to --checkpoint doesn't existresume fails and reports the missing path.
--checkpoint is omittedThe most recent valid checkpoint in save_dir is used automatically.

Once resumed, AutoCheck's checkpoint loop behaves exactly as it did before the interruption — same interval, same max_session_time, same notification settings — so you don't need to change anything else in your config.

See Configuration for everything under checkpoint, and the Getting Started guide for how save_dir is laid out on disk.
