Limitations

AutoCheck is under active development as a graduation project. This page is an honest account of what the current version doesn't yet do — so you know what to expect before you build on it.

No distributed or multi-node training

AutoCheck currently supervises single-process workloads running on a single machine. It doesn't support jobs that span multiple processes, multiple nodes, or multiple GPUs. Reliable checkpoint-and-resume across a coordinated, multi-machine training run isn't guaranteed by the current version — this is left for future work.

If your training script launches multiple worker processes or uses a multi-GPU distributed backend, AutoCheck is not yet a fit for that setup.

Java and C/C++ support is shallower than Python

All four supported languages — Python, Java, C, and C++ — share the same core checkpointing engine; only the tracing mechanism used to discover and capture program state differs between them. That said, the depth and granularity of automatic state discovery is not equal across languages.

Python benefits from the language's dynamic nature, which makes non-invasive state discovery considerably easier. Java and C/C++ have more limited automatic discovery by comparison — for HPC jobs in these languages, you explicitly list the variables to track via hpc.tracked_states rather than relying on full automatic discovery. See Language Support for what each language currently requires.

Separately, the deeper ML-framework integrations — PyTorch, Keras/TensorFlow, and Scikit-learn — are Python-only. There is no equivalent framework-level checkpointing for machine learning libraries in other languages.

Debugger-based tracing has overhead and requirements

C and C++ programs are traced through a debugger rather than native language hooks, which introduces some runtime overhead compared to Python's tracing approach. It also means your target program must be compiled with debug information (e.g. -g) for AutoCheck to reliably resolve and read tracked variable names — a stripped release binary won't work. See the C and C++ examples for the compile step.

No GPU or hardware-level state capture

AutoCheck checkpoints process-level and application-level state — model weights, optimizer state, and the variables you track — but it does not capture low-level hardware state. This means things like in-flight GPU kernel execution or device memory that isn't exposed through your framework's or adapter's API are outside what gets saved and restored.

Local storage only

Checkpoints are persisted to local or directly attached storage via checkpoint.save_dir. Remote or cloud-based storage backends — writing checkpoints directly to S3, GCS, or similar — are not yet implemented.

This is a snapshot of the current build

These limitations describe the scope of the current implementation, not the long-term design. Several — particularly distributed training and remote storage — are explicitly called out as candidates for future work beyond this graduation project.

For what AutoCheck does support today, see Language Support, Configuration, and the per-language examples for Python, Java, C, and C++.
