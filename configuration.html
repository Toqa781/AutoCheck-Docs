Configuration System

All checkpointing behavior is controlled through a single config.yaml file — no environment variables, no scattered flags, no code modifications required.

Which config do I need?

Use the table below to find your starting point. All examples are complete, minimal, and ready to copy.

Single source of truth

Every parameter that affects how, when, and where AutoCheck saves state lives in one config.yaml. This keeps a run reproducible: the config file alongside the checkpoint directory is enough to understand exactly how that run was supervised.

Python ML example

For a PyTorch training job. Replace pytorch with tensorflow for Keras.

yaml
language: python

checkpointing:
  enabled: true
  interval_minutes: 30
  session_time_limit: 120
  storage_path: ./checkpoints
  framework: pytorch

training:
  epochs: 100
  batch_size: 64

Java HPC example

For a Java program. AutoCheck uses JDWP (port 5005) to read state and a control socket (port 5006) to inject restored values — both must be free before the run starts.

yaml
language: java

java:
  main_class: HPCJob
  classpath: ./build
  jdwp_port: 5005
  control_port: 5006

hpc:
  enabled: true
  scheduler: jdwp

checkpointing:
  enabled: true
  interval_minutes: 30
  session_time_limit: 120
  storage_path: ./checkpoints

C / C++ HPC example

For C and C++ binaries. AutoCheck uses ptrace-based tracing — no JDWP ports needed.

yaml
language: c          # or cpp

hpc:
  enabled: true

checkpointing:
  enabled: true
  interval_minutes: 30
  session_time_limit: 120
  storage_path: ./checkpoints

Where AutoCheck looks for the file

The config path is passed as the first argument to RunnerScript. If omitted, AutoCheck defaults to ./config.yaml in the current working directory.

bash
# explicit path
python runner.py --config /path/to/my_config.yaml

# defaults to ./config.yaml
python runner.py
