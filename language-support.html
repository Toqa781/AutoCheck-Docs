Language Support

AutoCheck supervises jobs written in Python, Java, C, and C++ without requiring any changes to your source code. This page covers what each language needs and how to point AutoCheck at it.

Supported languages at a glance

Set system.language in your config.yaml to match your target program. AutoCheck uses this value to choose how it attaches to and reads state from your process — you never need to import anything or call a "save" function yourself.

One engine, four languages

The same checkpointing engine — triggers, intervals, save/resume logic, notifications — works identically regardless of language. Only the mechanism AutoCheck uses to read your program's state changes.

Python

Use execution_mode: ml with language: python for PyTorch or TensorFlow training scripts. AutoCheck observes your script while it runs and captures model weights, optimizer state, and training progress automatically — you keep writing your training loop exactly as you always have.

yaml
system:
  execution_mode: ml
  language: python
  fram_schd: pytorch       # or tensorflow
  run_id: run1

checkpoint:
  program_path: Testing_phase/deepNN.py
  save_dir: checkpoints/DeepNN

Nothing further is required beyond having the framework itself installed. See the ML example in Configuration for the full file.

Java

Use execution_mode: hpc with language: java for long-running Java jobs. List the variables you want captured at each checkpoint under hpc.tracked_states — these should be the names of local or instance variables already present in your program, exactly as they appear in your source.

yaml
system:
  execution_mode: hpc
  language: java
  fram_schd: general
  run_id: run1

checkpoint:
  program_path: Testing_phase/HPCJob.java
  save_dir: checkpoints

hpc:
  tracked_states:
    - name: progress_percent
      type: float
      source: jdwp

Each entry under tracked_states needs a name, a type, and source: jdwp. You don't need to add any logging, annotations, or AutoCheck-specific code to your Java program — just make sure your JDK supports remote debugging, which is standard on any modern JDK install.

C and C++

Use execution_mode: hpc with language: c or language: cpp for compiled simulation or scientific computing jobs. As with Java, list the variables to capture under hpc.tracked_states.

yaml
system:
  execution_mode: hpc
  language: cpp          # or c
  fram_schd: general
  run_id: run1

checkpoint:
  program_path: Testing_phase/simulation
  save_dir: checkpoints

hpc:
  tracked_states:
    - name: iteration_count
      type: int
      source: gdb

Two things to check before running a C or C++ job:

  • GDB must be installed and reachable on your PATH — AutoCheck uses it to attach to your running binary.
  • Compile with debug symbols (e.g. -g with gcc/g++) so the variable names in tracked_states can actually be resolved. A stripped release binary won't expose the names AutoCheck needs.

checkpoint.program_path should point at your compiled binary, not source code.

Which mode goes with which language?

In short: Python pairs with execution_mode: ml for training jobs, while Java, C, and C++ pair with execution_mode: hpc for everything else. See Which config do I need? for the full decision table, and the schema reference for every field under hpc.tracked_states.
