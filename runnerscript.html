RunnerScript Orchestrator

The entry point that bootstraps the execution environment and the user's training script — "the bridge" between the CLI and the controller.

Relevant source files runnerscript/__init__.py main.py

Bootstrap lifecycle

RunnerScript is responsible for getting from "a config file and a model" to "a supervised, checkpointed training run." It does this in a fixed sequence:

  1. Load and validate the configuration via ConfigManager
  2. Construct the appropriate StateTracker for the job type (ML or HPC)
  3. Construct the framework-specific CheckpointManager (PyTorch, Keras, or generic)
  4. Start the AutonomousController on the main thread
  5. Launch the user's training call on a dedicated daemon thread

train_with_checkpoints

The primary public method. It accepts the objects your training loop already has, model, dataloader, optimizer, and wires them into the supervision chain without requiring any change to how they're used inside the loop itself.

python
from autocheck import RunnerScript

runner = RunnerScript(config_path="config.yaml")
runner.train_with_checkpoints(model, dataloader, optimizer)
ParameterTypeDescription
modelframework model objectThe model whose weights will be captured at each checkpoint.
dataloaderiterablePassed through untouched; AutoCheck only observes iteration progress.
optimizerframework optimizerOptimizer state is captured alongside model weights.
Your training loop is unmodified

Internally, RunnerScript hands these objects to a TraceLayer instance, which observes their state from outside the loop via sys.settrace rather than requiring explicit save calls.
